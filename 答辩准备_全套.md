## 一、快速概览

> 如果老师让你"简单介绍一下整个项目"，用下面这段话（约1分钟）：

我们做的是一个桌面端的算法可视化教学平台，叫 AlgoVisual。技术栈是 Electron + React + TypeScript。核心功能是把数据结构教材里 55 个算法全部做成可交互的动画——学生可以选算法、点播放、看每一步数据怎么变化，还能自己输入数据、调速度。

系统分成 5 层架构：最底层是类型定义，往上是算法引擎层（负责生成动画步骤），再往上是状态管理层（Zustand）、可视化层（5 种渲染组件）、最顶层是用户界面（三栏 IDE 布局）。

我们5个人分工：我负责整体架构和类型系统，王林镟负责55个算法的步骤生成，成员C负责5种可视化组件的动画渲染，成员D负责UI界面和交互逻辑，成员E负责文档、测试和打包。最终实现了 55 个算法、7 大分类、macOS 和 Windows 双平台安装包。TypeScript 零类型错误，生产构建 398KB。

在开发过程中深度使用了 AI 辅助——主力是 Claude Code（Claude Opus 4.6），辅助是 GitHub Copilot。约 60 轮对话，生成约 6000 行代码，人工修改约 20%。

## 二、各模块深讲与问答

### 成员A：项目架构 & 核心框架（聂琦钢）

**负责内容**：Electron 脚手架、TypeScript 类型系统、Zustand 状态机、项目配置、数据管理子系统、跨平台打包

#### 讲解要点

**1. 5层分层架构**

我把系统分成 5 层，每一层只依赖下一层：
- L5 类型层：`types/algo.ts`，定义 VisualStep、AlgorithmMeta 等核心接口
- L4 引擎层：`registry.ts`，55 个算法的步骤生成器
- L3 状态层：`AlgoContext.tsx`，Zustand Store
- L2 可视化层：`Canvas.tsx` + 5 种 Visualizer
- L1 表示层：Sidebar/ControlBar/CodePanel/LogPanel/DataControl

这样设计的好处：算法逻辑是纯 TypeScript 函数，不依赖 React；UI 组件只从 Zustand 读状态，不直接调算法。两层之间靠接口通信，互不影响。

**2. VisualStep 接口设计**

这是整个系统最核心的接口。核心字段是 `data`（当前快照）、`activeIndices`（操作中的索引，黄色的）、`markedIndices`（已确定的，绿色的）、`actionType`（操作类型，驱动动画）。扩展字段比如 `graphEdges`、`gridData` 是可选的，只有特定可视化类型才用到。

设计原则有三条：完整性（每步独立快照，支持单步后退）、通用性（排序/树/图/迷宫统一描述）、动画友好（16 种类型对应 16 种动效）。

**3. 状态机设计**

四个状态：idle → playing → paused → completed。核心方法：`play()` 启动 setInterval，每 `1000/speed` ms 调一次 `stepForward()`；`stepForward()` 切到下一步，到了末尾自动切 completed。Zustand 的 `getState()` 在 timer 里安全读取最新状态，不会闭包陷阱。

**4. 随机数据生成**

为 55 个算法分别定制了生成规则。排序生成随机数组、查找生成有序数组+存在的目标值、图算法生成随机节点和边权重、N皇后限制在 4-8、中缀表达式生成合法数学式。不能一刀切全用随机数组。

**5. 打包踩坑**

- Electron 二进制用 ELECTRON_MIRROR 镜像下载
- 版本号必须精确（^30.5.1 → 30.5.1）
- winCodeSign/nsis/wine 下载多次超时，重试 3-5 次成功
- Mac dmg 92MB，Windows exe 74MB

#### 高频问答

**Q：为什么选 Electron 而不是 Tauri？**
Tauri 更轻量但生态不如 Electron 成熟，M1 兼容性当时还有问题。团队有 React 基础，Electron 的 Chromium 天然兼容 Web 技术栈。

**Q：为什么选 Zustand 而不是 Redux？**
Redux 太重，Context 会导致不相关组件重渲染。Zustand 的 `getState()` 可以在 setInterval 里安全读状态，播放引擎就靠这个。

**Q：新增一个算法要改哪些文件？**
不改任何文件。调用 `registerAlgorithm({ meta, generateSteps })` 就行。侧边栏、Canvas、CodePanel 全部自动适配。

**Q：类型系统扩展时遇到过什么问题？**
ActionType 从 6 种扩展到 16 种时，所有 `Record<ActionType, ...>` 都报类型错误。解决方案是逐个补全映射。TypeScript 的 exhaustiveness check 精确告诉你了漏了哪个。

**Q：AI 在这个模块帮了什么？**
生成类型接口框架、Zustand Store 代码、55 条随机数据规则的分支逻辑、electron-builder 配置调试。人工修改约 20%——主要在随机数据的约束规则和打包配置的网络问题处理。

### 成员B：算法引擎（王林镟）

**负责内容**：算法注册表架构、全部 55 个算法的 `generateSteps` 函数

#### 讲解要点

**1. 算法注册表模式**

`registerAlgorithm(entry)` 把算法注册到全局 Map，`getAlgorithmById(id)` 查询。55 个算法在 `registry.ts` 顶层自动注册。新增算法只需：定义 meta → 写 generateSteps → 调 registerAlgorithm。

**2. generateSteps 的设计规范**

- 每一步必须输出完整数据快照（`[...arr]`），不能是增量
- `activeIndices` 指向操作中位置，`markedIndices` 指向已完成位置
- `highlightLines` 必须与 templateCode 行号对应
- `description` 用中文，格式是"动作+对象+结果"
- 纯函数，不修改输入，不依赖外部状态

**3. 典型算法举例**

**冒泡排序**：外层 i 循环，内层 j 循环比较相邻元素。每次比较输出一个 compare 步骤，需要交换时输出一个 swap 步骤。每轮结束把末尾元素标记为 marked。时间复杂度 O(n²)，步骤数约 45。

**快速排序**：选 pivot，遍历分区，比 pivot 小的放左边。每个分区操作输出 compare/swap 步骤。递归过程用 divide/merge 标记。

**Dijkstra**：维护 dist 数组和 visited 数组。每次选最小距离未访问节点，松弛所有邻居边。松弛成功输出 relax 步骤，边高亮通过 graphEdges 传递。

**N皇后**：回溯搜索。每次放置皇后输出 insert，冲突跳过输出 highlight，撤回输出 backtrack。第一个解找到就停，最后统计总解数。

**BST 插入**：层序数组存储，`while (idx < bst.length && bst[idx] !== 0)` 找到空位。值为 0 的是空槽，遇到空槽就停——这是和完全二叉树的关键区别。

**4. 用户自定义参数的实现**

查找类算法：`data` 末尾位 = target，`data.slice(0,-1)` = 数组。线性表插入：末尾两位 = [插入值, 插入位置]。图算法：`[N, from,to,w, ...]` 边列表格式。

#### 高频问答

**Q：55 个算法每步都正确是怎么保证的？**
每个算法写完都用默认数据和自定义数据各跑一遍，核对每一步的 data/activeIndices/markedIndices。TypeScript 类型检查是一道防线，人工验证是第二道。边界情况（单元素、全逆序、全有序）也测过。

**Q：为什么步骤不能是增量（patch），而必须是完整快照？**
单步后退需要完整快照。如果是增量，后退一步需要反向计算——比如 swap 的逆操作也是 swap 还好说，但 BST 插入的逆操作就很复杂。完整快照直接读 `steps[currentStep - 1]`。

**Q：算法步骤数会不会太多？**
控制在合理范围。排序类 20-100 步，图类 15-40 步，DP 类只输出状态变化的步骤（不是每个单元格都单独一步）。N皇后只展示找到第一个解的过程。

**Q：你是怎么处理树算法的？**
树算法用层序数组存储（index i 的左子=2i+1，右子=2i+2）。空节点值为 0，TreeVisualizer 跳过渲染。BST 插入时注意 `bst[idx] === 0` 表示空槽，循环应该在这里停，不能继续比较——这是你之前花了大功夫修的 bug。

**Q：AI 在算法生成中帮了什么？**
每个算法的框架代码由 AI 生成，然后人工调整边界条件、中文描述和行号映射。比如冒泡排序，AI 生成的结构是对的，但内层循环的上限 `n-1-i` 和 markedIndices 的标记时机需要人工确认。平均每个算法 3-5 轮迭代。

### 成员C：可视化引擎

**负责内容**：Canvas 编排器、5 种 Visualizer 组件、Framer Motion 动画系统

#### 讲解要点

**1. 5 种可视化组件**

| 组件 | 适用场景 | 渲染方式 | 关键动画 |
|------|---------|---------|---------|
| ArrayVisualizer | 排序/查找/栈/队列 | 垂直柱状图（排序）或水平管道（栈队列） | compare 脉冲+发光, swap Y轴跳跃[-32px], insert 缩放生长 |
| LinkedListVisualizer | 链表/链栈/链队列 | SVG 矩形节点 + 箭头连线 + null 终止符 | 节点 stagger 入场, active 发光边框 |
| TreeVisualizer | 树遍历/BST/堆/哈夫曼 | SVG 圆圈 + 递归区域分配布局 | 层级分隔线+标签, 边 opacity 淡入 |
| GraphVisualizer | BFS/DFS/Dijkstra/Prim/Kruskal | N 节点环形布局 + 边权重 badge | 松弛边发光+旋转虚线环 |
| GridVisualizer | 迷宫/N皇后/FloodFill/A*/DP | CSS Grid 二维单元格 7 色状态 | scale/opacity 变化 |

**2. 树布局算法**

核心算法是递归子树区域分配：`x = (leftBound + rightBound) / 2`。左子占据 `[leftBound, parentX]`，右子占据 `[parentX, rightBound]`。空节点跳过不渲染。无论树是平衡还是偏斜，都能产生自然树形，节点位置稳定不跳变。

这个算法经历了 4 次迭代：均匀分布（偏斜树全堆在一边）→ 遮罩法（初始状态叶子散落深层，内部节点为零导致无法遍历）→ 最终递归区域分配。

**3. 图布局算法**

N 个节点均匀分布在圆周上：`angle = 2πi/n - π/2`。边权重以圆角 badge 显示在边中点偏移处（`offset = midpoint + perpendicular × 18px`）。松弛边高亮用 SVG filter 发光效果。当前节点用旋转虚线环指示。

**4. 动画引擎参数**

16 种 ActionType 对应不同 Spring 参数：

| ActionType | stiffness | damping | 效果 |
|-----------|-----------|---------|------|
| compare | 300 | 20 | 轻微脉冲 + 发光 |
| swap | 260 | 18 | Y轴跳跃 [-32px] |
| move | 280 | 22 | 平滑滑动 |
| insert | 300 | 18 | 从 0 缩放生长 |
| delete | 200 | 20 | 缩小到 0 + 淡出 |
| complete | 260 | 15 | 全绿脉冲 |

速度档位控制的是步骤间隔（0.25x→4000ms, 0.5x→2000ms, 1x→1000ms, 2x→500ms, 4x→250ms），动画本身的 duration 不受速度影响。

**5. 字符直输的 showChars 机制**

括号匹配和中缀转后缀需要显示字符标签。但柱形图的数字标签对所有算法都生效——排序算法里 42 也会显示为 `'*'`（ASCII 42=*）。所以加了一个 `showChars` prop，从 Canvas 判断算法 ID 传入，只有这两个算法才显示字符标签。

#### 高频问答

**Q：为什么不用 Canvas API 统一渲染？**
SVG 适合树和图（需要画连线），DOM 适合数组和链表（需要弹性布局），CSS Grid 适合迷宫和 DP 表。每种场景用最合适的方案，比强制统一效果好。

**Q：树布局算法是怎么想出来的？**
不是从哪抄的。前三个版本都失败了，分析原因后从"分配空间"的角度重新思考——每个节点把自己拥有的水平空间分给左右子树，递归下去就是自然的树形。

**Q：Framer Motion 和其他动画方案比有什么优势？**
声明式 API（`animate={{ y: [0, -32, 0] }}`），Spring 物理引擎天然有弹性和阻尼感，`layout` 属性自动处理元素位置过渡。CSS Animation 做不到这么细粒度的每步控制。

**Q：图布局怎么适配不同节点数？**
环形布局公式 `angle = 2πi/n - π/2` 天然支持任意 n。节点标签用 ABCD...（最多 16 个，因为 `NODE_LABELS = 'ABCDEFGHIJKLMNOP'`）。边权重 badge 需要偏移计算避免和边重叠。

**Q：AI 在可视化中帮了什么？**
生成 Framer Motion 动画框架、颜色映射表。但动画参数调优（stiffness/damping）是手工试出来的，树布局核心算法是手工推导的。AI 生成约 1500 行，人工修改约 25%。

### 成员D：交互界面 & 教学功能

**负责内容**：Layout/Sidebar/ControlBar/CodePanel/LogPanel/DataControl，共 7 个 UI 组件

#### 讲解要点

**1. 三栏 IDE 布局**

```
ControlBar (h=48px)          ← 播放控制
├── Sidebar (w=260px)        ← 7分类算法目录
├── Canvas (flex-1)          ← 可视化画布
└── Right (w=420px)          ← 代码面板 + 日志面板
DataControl (h=40px)         ← 数据管理
```

弹性布局，最小窗口 1100×700。侧边栏和右侧面板固定宽度，画布自动填充剩余空间。

**2. 代码面板的语法高亮器**

自研 `tokenizeLine` 词法分析器（约 60 行），零外部依赖。识别 8 类 token：keyword（紫）、string（绿）、number（橙）、comment（灰）、operator（青）、function（蓝）、type（黄）、variable（浅灰）。关键字列表：function/const/let/var/return/if/else/for/while/class/new/this/null/undefined/true/false。

处理了字符串引号内符号误识别的问题——用引号状态机，遇到引号切换"在字符串内/外"状态。

**3. 播放控制与键盘快捷键**

Space 播放/暂停，← 单步后退，→ 单步前进，R 重置。输入框焦点时不触发（检查 `target instanceof HTMLInputElement`）。进度条有分段彩色圆点——每个步骤一个 3px 圆点，按 actionType 着色，已走过的亮色、未走过的灰色，当前步骤放大 1.8 倍。

**4. 数据控制台的多模式输入**

支持 4 种输入模式：数字数组（逗号分隔）、单个数字（N皇后/棋盘大小）、字符直输（括号/表达式，自动转 charCode）、边列表（图算法，`N,from,to,w,...`）。输入提示按算法类型变化。`setCustomData` 后自动调 `loadAlgorithm` 重生成步骤。

**5. 日志面板的操作统计**

在"执行历史"上方，统计 compare/swap/move/insert/delete 各操作的出现次数，按颜色区分显示。数据来源是 `logs.filter(l => l.actionType === t).length`。

#### 高频问答

**Q：为什么要自研语法高亮器而不用 Prism.js？**
零依赖 + 可控。Prism.js 是一个完整的库，我们只需要对已知的模板代码做高亮，不需要支持所有语言。自研的 tokenizeLine 只有 60 行，够用且轻量。

**Q：数据输入怎么适配这么多种格式？**
统一用逗号分隔解析。如果检测到非数字字符（`/[^0-9,\s\-]/`）且算法是括号匹配或中缀转后缀，自动按字符模式处理（每字符 → charCode）。图算法检测到超过 3 个数字且第一个是正整数，按边列表处理。

**Q：进度条圆点为什么限制 40 个？**
步骤数可能上百，全显示的话圆点会密集到看不清。取前 40 步采样，用 `left: ${(i+1)/totalSteps*100}%` 按比例分布，视觉效果能代表整体进度。

**Q：AI 在 UI 中帮了什么？**
生成组件框架和事件处理、tokenizeLine 核心逻辑。人工补充了 8 类 token 的颜色映射、键盘快捷键的焦点检测、多模式输入解析。AI 生成约 1200 行，人工修改约 20%。

#### 高频问答

**Q：测试了多少个算法？**
全部 55 个都跑了。用默认数据跑一遍，随机数据跑一遍，自定义数据跑一遍。发现的 bug（大概 30+ 个）反馈给对应成员修复。

BST 查找：目标值混入了树数组导致查找路径错误。还有算法数据不生效的问题——`setCustomData` 没调 `loadAlgorithm`，输入完数据画面不更新。都是测试中发现的。

**Q：Mermaid 图为什么有兼容性问题？**
Mermaid 不同版本对语法的支持程度不一样。mindmap 是 9.3+ 才有的，stateDiagram-v2 在一些在线渲染器中报解析错误。解决方案是选择兼容性最好的语法（graph/flowchart/sequenceDiagram/erDiagram），并在 mermaid.live 上逐个验证。

**Q：文档编写花了多少时间？**
需求文档约 2 天，设计文档约 2 天，最终项目文档约 1 天。Mermaid 图是 AI 生成框架 + 人工修正语法兼容性，比纯手画效率高很多。

## 三、跨模块综合问题

> 这些问题涉及多个模块，可能会问任何一个成员。

### Q1：整个项目你觉得哪个部分最有技术含量？

**树布局算法**和 **VisualStep 接口设计**。树布局经历了 4 次迭代才找到正确的算法，不是抄现成的。VisualStep 接口要能同时描述排序的数组比较、图的最短路径松弛、迷宫的墙壁挖掘——这种"统一性"的设计是最难的。

### Q2：如果让你重新做这个项目，你会怎么做？

1. 先用更少的时间做接口设计——第一版 VisualStep 只有 6 个核心字段，后来扩展到 10+，如果能一开始就考虑周全，后续扩展成本更低
2. 哈夫曼树从一开始就设计成"森林→单一树"的渐进可视化，而不是后来返工
3. 提前配置 CI/CD，避免本地打包受网络影响

### Q3：你觉得这个项目最大的创新点是什么？

**统一 VisualStep 接口 + 算法注册表模式 + 差异化动画引擎**三者结合。不是算法本身有多新（算法都是经典算法），而是把 55 个算法统一到一个可扩展的架构里，新加算法零改动，这个设计思路是创新的。

### Q4：AI 生成了那么多代码，你怎么保证质量？

三道防线：TypeScript 类型检查（编译时拦截）→ 人工逐算法验证步骤正确性 → 边界测试。AI 生成的代码第一次正确率约 70%，需要 3-5 轮迭代。最后人工修改了约 20%。

### Q5：你们的代码怎么管理的？有没有版本控制？

项目规模不大（~3000 行），主要通过文件分工来避免冲突——5 个人负责不同的目录。类型接口作为"合约"先行定义，大家各自开发互不影响。

### Q6：项目有没有考虑到无障碍？比如色盲用户？

目前没有专门做无障碍适配。我们用了 Framer Motion 的 `useReducedMotion` 适配了减少动画的偏好，CSS 里也写了 `prefers-reduced-motion` 的降级方案。activeIndices 和 markedIndices 除了颜色还有大小和边框的差异化，不完全依赖颜色区分。
